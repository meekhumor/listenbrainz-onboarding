## Overview

[ListenBrainz](https://github.com/metabrainz/listenbrainz-server) is an open source music listening tracker built and operated by the MetaBrainz Foundation. It records what you listen to, generates personal statistics and builds a social graph of listeners from all over the world.

During this summer, I designed and shipped a full stack, cross page user onboarding system that guides new users through the platform's core features via guided tours. 

The primary challenge was building guided tours that continue seamlessly across route transitions, this was the problem which [React Joyride](https://react-joyride.com/) does not solve natively. The solution required a understanding of React's rendering pipeline, the browsers DOM mutation lifecycle, and joyride's internal state machine.


## Problem Statement

### The Problem

A brand new ListenBrainz user who logs in for the first time is dropped into an empty dashboard. Their listen history is empty. Their stats charts show no data. Their social graph is unconnected.

ListenBrainz's feature set is rich but non obvious to new user. Discovering that you can connect Spotify to import your listening history, or that there is an artist origin world map in the stats section, requires deliberate exploration that most new users will not undertake in first place. 

### Requirements

The onboarding system need to satisfy these four things:

- **Cross route continuity.** A single tour must spotlight elements on different pages without losing its place.
- **Session persistence.** If a user closes the browser mid tour, they resume from exactly where they left off.
- **Progressive unlocking.** Feature tours only appear once the user has meaningful data to show (otherwise what's the point of showing empty components).
- **Central control.** Users need a single place to start, restart, or track all available tours.

### What existed before

ListenBrainz had no onboarding system of any kind. No welcome modal, no guided tour.

## Design Plan

The design plan was as follows:

- A **Welcome Modal** on first login with two choices: Skip or Start Tour.
- A **Setup Tour** crossing four routes: dashboard → settings → music services → add data.
- Three **Feature Tours** (Listens, Stats, Social) which will unlock progressively based on data availability.
- A **Feature Guide** page for starting, resuming, or restarting tours.
- A **MutationObserver** strategy for DOM synchronization during route transitions.
- A database table (`user_onboarding_state`) for progress tracking.
- Two  API endpoints for reading and writing onboarding state.

## Implementation Overview

The implementation is organized as follows:

1. **Database schema and migration.** New table `user_onboarding_state` with composite primary key, foreign keys, indexes, and a migration script.
2. **Backend API.** Flask blueprint with GET and POST endpoints which will fetch new state, evaluate unlock condition and update progress respectively.
3. **Frontend tour infrastructure.** Integration with `GlobalAppContext`, `useTour` hook, `waitForElement`and `MutationObserver` utility.
4. **Four interactive guided tours.** SetupTourManager, ListensTourManager, StatsTourManager, SocialTourManager.
5. **Feature Guide.** The central control page with start, resume and restart tour feature.


## Database

### Schema

```sql
CREATE TABLE user_onboarding_state (
  user_id      INTEGER     NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
  tour_id      VARCHAR(50) NOT NULL,
  status       VARCHAR(20) NOT NULL DEFAULT 'not_started',
  current_step INTEGER     NOT NULL DEFAULT 0,
  unlock_ready BOOLEAN     NOT NULL DEFAULT FALSE,
  unlocked_at  TIMESTAMP,
  updated_at   TIMESTAMP   NOT NULL DEFAULT NOW(),
  PRIMARY KEY (user_id, tour_id)
);
```

### Relational Model

| Column | Type | Description |
|--------|------|-------------|
| `user_id` | `INTEGER` | Foreign key to user table |
| `tour_id` | `VARCHAR(50)` | Tour identifier |
| `status` | `VARCHAR(20)` | Tour status |
| `current_step` | `INTEGER` | Index of the last active step |
| `unlock_ready` | `BOOLEAN` | Whether the tour is available |
| `unlocked_at` | `TIMESTAMP` | When the unlock condition was first satisfied |
| `updated_at` | `TIMESTAMP` | Last modification timestamp |

### Primary key 

The composite primary key `(user_id, tour_id)` has exactly one row per user per tour at the database level.

### unlock_ready flag

`unlock_ready` is a write once flag: once set to `TRUE`, it is never updated again for that (user, tour) pair since server never runs unlock condition queries for already unlocked tours, reducing DB load on every GET request.

## Backend

### GET /1/user/[username]/onboarding

**Purpose:** Returns the complete onboarding state for all tours. On every request, the server queries the database and checks whether the user has met unlock conditions for any tours that are not yet unlocked.

Response shape:

```json
{
  "setup":   { "status": "completed",   "current_step": 6, "unlock_ready": true  },
  "listens": { "status": "in_progress", "current_step": 2, "unlock_ready": true  },
  "stats":   { "status": "not_started", "current_step": 0, "unlock_ready": false },
  "social":  { "status": "not_started", "current_step": 0, "unlock_ready": false }
}
```

### POST /1/user/[username]/onboarding

**Purpose:** Update the progress of a specific tour. Accepts a payload specifying `tour_id`, `status`, and `current_step`, and updates the user's database record accordingly.

### Unlock condition evaluation

Unlock requirements are evaluated server side to guarantee integrity:
- **Listens Tour:** Unlocks once the user has recorded at least 1 listen in the `listen` table.
- **Stats Tour:** Unlocks when the user has generated at least 6 distinct week range stats (e.g., `listening_activity`, `artists`, `release_groups`, `recordings`, `daily_activity`, `artist_map`) in the `user_data` table.
- **Social Tour:** Unlocks once the user has followed at least one other user and has at least one entry in the recommendations `similar_user` table.

Once a condition is met, the `unlock_ready` flag is written as `true`. 

The setup tour is pre unlocked for all new users because it has no data prerequisite, it guides users through connecting services before any data exists.

## Frontend

### GlobalAppContext integration

`onboardingState` is added to the existing `GlobalAppContext`. It is loaded once on application mount via a single API call and shared to all `TourManager` instances, avoiding prop drilling and duplicate API requests.

### The useTour hook

`useTour` is the central state machine managing React Joyride control state (such as `run`, `stepIndex`, and `paused`), local storage synchronization, API synchronization and cross page navigation.

### Local storage and API synchronization

- **Local storage** is written synchronously on every step change and it is read on mount for instant restoration.
- **The API** is written asynchronously. It is the source of truth on initial page load and persists across devices.

## Cross Page Navigation

Cross-page navigation is main challenge of this project. React Joyride is designed to spotlight elements that exist in the current DOM. It has no built in mechanism for navigating to a different route and waiting for new elements to mount before resuming.

When a user triggers a route transition, React Router mounts and renders the target page components asynchronously. Because target elements do not exist in the DOM at the moment of navigation, Joyride will error or render tooltips incorrectly.

### Alternatives considered

| Approach | Why rejected |
|----------|--------------|
| `setTimeout` with fixed delay | Not suitable since delay varies by device speed and network conditions. |
| `setInterval` polling | Wastes CPU cycles it is also not event driven. |
| `requestAnimationFrame` loop | Equivalent to polling but faster, it still wastes cycles. |
| **MutationObserver** | Event driven which fires exactly when DOM changes. |

### The MutationObserver solution

It observes the `document.body` for subtree alterations (`{ childList: true, subtree: true }`) to detect nested components.
- 10 second timeout automatically disconnects the observer and rejects the Promise if the element fails to render.
- The observer immediately disconnects once the target element is resolved.

### Complete cross page navigation sequence

1. User clicks "Next" on the current step. Joyride fires its callback.
2. `useTour` detects that the next step requires a different route.
3. Joyride is paused (`run` state is set to `false`) to avoid positioning errors.
4. Navigation is triggered via React Router.
5. The `MutationObserver` begins watching the DOM for the next step's target element.
6. The target page mounts, the element is detected, the observer disconnects, and the Promise resolves.
7. Joyride is resumed with the new step index, and the viewport scrolls to the spotlighted element.

## Feature Guide

### What it is

The Feature Guide is a page at `/settings/feature-guide/` that acts as a persistent control center for all available tours. Users see which tours they have completed, which are in progress, and which are still locked.

### The decoupling problem

Feature Guide and TourManager components are in completely separate React component trees. They cannot pass props or callbacks to each other directly.

### Event registry

| Tour | Start | Restart | Resume |
|------|-------|---------|--------|
| Setup | `lb:start-setup-tour` | `lb:restart-setup-tour` | `lb:resume-setup-tour` |
| Listens | `lb:start-listens-tour` | `lb:restart-listens-tour` | `lb:resume-listens-tour` |
| Stats | `lb:start-stats-tour` | `lb:restart-stats-tour` | `lb:resume-stats-tour` |
| Social | `lb:start-social-tour` | `lb:restart-social-tour` | `lb:resume-social-tour` |

Communication is established using custom browser events (`window.dispatchEvent` / `window.addEventListener`), decoupling the Feature Guide layout from the individual tour manager instances without global state libraries or prop drilling.

### Custom tooltip component

`CustomTooltip.tsx` replaces joyride's default tooltip.

## Challenges Faced

### 1. Cross page routing
React Joyride spotlights DOM elements. When a step targets an element on a different route, that element doesn't exist in the DOM yet.
**Solution:** The pause → navigate → MutationObserver → resume the run.

### 2. DOM subtree observation timing
React's async rendering which means the route wrapper mounts before its children. Observing only direct children misses deeply nested tour targets.
**Solution:** `{ childList: true, subtree: true }` on the MutationObserver. The callback runs `querySelector` on every mutation which is fast because native selector matching is implemented in C++ in the browser engine.

### 4. Viewport responsive tour targets
The Settings link is in the sidebar on desktop but behind a hamburger menu on mobile. Same tour but different targets.
**Solution:** `useViewportBreakpoints.ts` + `useMemo` rebuilds the steps array whenever the breakpoint changes.

### 5. Tour interruption handling
If a user navigates away by clicking a link (not the Joyride "Next" button), the tour state becomes stale.
**Solution:** A React Router navigation listener detects unexpected route changes. On unexpected navigation, the tour is paused and step index is persisted in database. TourModal then prompts resume on return.

### 6. Auto opening dropdowns
The "Add listens" dropdown is closed by default. Joyride cannot spotlight elements inside a closed dropdown.
**Resolution:** An `onBeforeRun` hook fires before Step 5 and clicks the dropdown toggle.
