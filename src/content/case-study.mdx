## Overview {#overview}

ListenBrainz is an open-source music listening tracker built and operated by the MetaBrainz Foundation. It records what you listen to, generates personal statistics, builds a social graph of listeners with similar taste, and integrates with streaming services via BrainzPlayer.

During my Summer 2026 internship, I designed and shipped a full-stack, cross-page user onboarding system that guides new users through the platform's core features via interactive guided tours. The system spans the entire product stack: a new PostgreSQL table, two Flask REST endpoints, a React hook state machine, a MutationObserver-based DOM synchronization utility, four distinct guided tours powered by React Joyride, a Feature Guide hub, and custom SCSS stylesheets.

The primary engineering challenge was building guided tours that continue seamlessly across React Router route transitions — a problem React Joyride does not solve natively. The solution required a careful understanding of React's rendering pipeline, the browser's DOM mutation lifecycle, and Joyride's internal state machine.

<Callout type="important">
This document reflects the final shipped implementation. Where the original design document differed from what was built, the implementation takes precedence.
</Callout>

## Problem Statement {#problem-statement}

### The cold-start problem

A brand-new ListenBrainz user who logs in for the first time is dropped into an empty dashboard. Their listen history is empty. Their stats charts show no data. Their social graph is unconnected. The BrainzPlayer at the bottom of the screen is inert.

ListenBrainz's feature set is rich but non-obvious. Discovering that you can connect Spotify to import your listening history, or that there is an artist origin world map in the stats section, requires deliberate exploration that most new users will not undertake unprompted. This is the classic cold-start problem in feature-rich applications.

### Requirements

The onboarding system needed to satisfy four constraints:

- **Cross-route continuity.** A single tour must spotlight elements on different pages without losing its place.
- **Session persistence.** If a user closes the browser mid-tour, they resume from exactly where they left off.
- **Progressive unlocking.** Feature tours only appear once the user has meaningful data to show.
- **Central control.** Users need a single place to start, restart, or track all available tours.

### What existed before

ListenBrainz had no onboarding system of any kind. No welcome modal, no guided tour, no progressive disclosure mechanism. This work was greenfield.

## Original Design {#original-design}

The design document specified the following system before implementation began:

- A **Welcome Modal** on first login with two choices: Skip or Start Tour.
- A **Setup Tour** crossing four routes: dashboard → settings → music services → add data.
- Three **Feature Tours** (Listens, Stats, Social) unlocking progressively based on server-side data conditions.
- A **Feature Guide** hub at `/settings/feature-guide/` for starting, resuming, or restarting tours.
- A **MutationObserver** strategy for DOM synchronization during route transitions.
- A `user_onboarding_state` **database table** for persistent progress tracking.
- Two **REST API endpoints** (GET and POST) for reading and writing onboarding state.

### Divergences from the design

Three implementation details diverged from the original specification:

- **60-second auto-resume threshold.** When a user clicks "Connect a Service" and is redirected to an OAuth provider, they leave the ListenBrainz domain. The implementation uses 60 seconds: returns within 60s → auto-resume; returns after 60s → resume modal.
- **Programmatic dropdown opening.** Step 5 of the Setup Tour spotlights items inside the "Add listens" dropdown. A custom `onBeforeRun` hook clicks the toggle before mounting the step.
- **Viewport-aware step targets.** The Settings link is in the left sidebar on desktop but behind a hamburger menu on mobile. `useViewportBreakpoints.ts` overrides step targets per breakpoint.

## Implementation Overview {#implementation-overview}

The implementation is organized into six work streams:

1. **Database schema and migration.** New table `user_onboarding_state` with composite primary key, foreign keys, indexes, and a migration script.
2. **Backend API.** Flask blueprint with GET (fetch + unlock evaluation) and POST (upsert progress) endpoints. Seeding logic on new user registration.
3. **Frontend tour infrastructure.** `GlobalAppContext` integration, `useTour` hook, `waitForElement.ts` MutationObserver utility, and viewport-aware step overrides.
4. **Four interactive guided tours.** SetupTourManager, ListensTourManager, StatsTourManager, SocialTourManager.
5. **Feature Guide hub and UI integrations.** The control center page, browser CustomEvent registry, and ID additions to existing components.
6. **Stylesheets.** Custom SCSS for the tooltip card, modals, and Feature Guide page.

## Database {#database}

### Schema

```sql
CREATE TABLE user_onboarding_state (
  user_id      INTEGER     NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
  tour_id      VARCHAR(50) NOT NULL,
  status       VARCHAR(20) NOT NULL DEFAULT 'not_started',
  current_step INTEGER     NOT NULL DEFAULT 0,
  unlock_ready BOOLEAN     NOT NULL DEFAULT FALSE,
  unlocked_at  TIMESTAMP,
  updated_at   TIMESTAMP   NOT NULL DEFAULT NOW(),
  PRIMARY KEY (user_id, tour_id)
);
```

### Column reference

| Column | Type | Description |
|--------|------|-------------|
| `user_id` | `INTEGER` | FK → `"user"(id)`. `ON DELETE CASCADE` removes rows when user deletes account. |
| `tour_id` | `VARCHAR(50)` | Tour identifier. Values: `setup`, `listens`, `stats`, `social`. New tours add rows — no schema change. |
| `status` | `VARCHAR(20)` | Tour state: `not_started` · `in_progress` · `skipped` · `completed`. |
| `current_step` | `INTEGER` | Zero-based index of the last active step. Used for exact-position resume. |
| `unlock_ready` | `BOOLEAN` | Whether the tour is available. Written exactly once per (user, tour) pair. |
| `unlocked_at` | `TIMESTAMP` | When the unlock condition was first satisfied. NULL until unlocked. |
| `updated_at` | `TIMESTAMP` | Last modification timestamp. Updated on every POST. |

### Primary key design

The composite primary key `(user_id, tour_id)` enforces exactly one row per user per tour at the database level. This eliminates the need for a surrogate key and makes upserts (`INSERT ... ON CONFLICT DO UPDATE`) straightforward without additional unique indexes.

### The write-once unlock flag

`unlock_ready` is a write-once flag: once set to `TRUE`, it is never updated again for that (user, tour) pair.

- The server never runs unlock condition queries for already-unlocked tours, reducing DB load on every GET request.
- There is no race condition between concurrent GET requests — the second write is semantically a no-op.

### Migration files

- `2026-07-05-create-user-onboarding-state.sql` — primary migration script
- `create_tables.sql`, `create_primary_keys.sql`, `create_indexes.sql`, `create_foreign_keys.sql` — additions to schema definition files

<Callout type="note">
The table lives in the main PostgreSQL database. Unlock conditions that query listen counts reach into the TimescaleDB listen table. Stats unlock conditions query the `user_data` table.
</Callout>

### Unlock condition evaluation

```python
def _check_unlock_conditions(user_id: int, states: dict) -> dict:
    if not states["listens"]["unlock_ready"]:
        has_listen = db.session.execute(
            text("SELECT 1 FROM listen WHERE user_id = :uid LIMIT 1"),
            {"uid": user_id}
        ).fetchone()
        if has_listen:
            _flip_unlock(user_id, "listens")
            states["listens"]["unlock_ready"] = True

    if not states["stats"]["unlock_ready"]:
        count = db.session.execute(
            text("""
                SELECT COUNT(DISTINCT stat_type) FROM user_data
                WHERE user_id = :uid
                  AND stat_type IN (
                    'listening_activity','artists','release_groups',
                    'recordings','daily_activity','artist_map'
                  )
                  AND stats_range = 'week'
            """),
            {"uid": user_id}
        ).scalar()
        if count >= 6:
            _flip_unlock(user_id, "stats")
            states["stats"]["unlock_ready"] = True

    if not states["social"]["unlock_ready"]:
        follows = db.session.execute(
            text("SELECT 1 FROM user_relationship WHERE user_0=:uid AND relationship_type='follow' LIMIT 1"),
            {"uid": user_id}
        ).fetchone()
        has_similar = db.session.execute(
            text("SELECT 1 FROM recommendation.similar_user WHERE user_id=:uid LIMIT 1"),
            {"uid": user_id}
        ).fetchone()
        if follows and has_similar:
            _flip_unlock(user_id, "social")
            states["social"]["unlock_ready"] = True

    return states
```

### Setup tour seeding

```python
# Called in provider.py on new user creation
def create_setup_onboarding_row(user_id: int):
    db.session.execute(
        text("""
            INSERT INTO user_onboarding_state (user_id, tour_id, status, unlock_ready)
            VALUES (:uid, 'setup', 'not_started', TRUE)
            ON CONFLICT DO NOTHING
        """),
        {"uid": user_id}
    )
    db.session.commit()
```

### The useTour hook

`useTour` is the central state machine for a single tour. Each TourManager instance owns one `useTour` instance. It manages:

- **Joyride control state:** `run`, `stepIndex`, `paused`
- **Local storage sync:** persists `stepIndex` for fast restore without waiting for API
- **API sync:** POSTs new step index and status after every Joyride callback
- **Cross-page navigation:** pause → navigate → observe → resume sequence
- **Viewport-aware step overrides:** replaces step targets per breakpoint
- **BrainzPlayer management:** hides player on conflicting steps
- **Auto-scroll:** calls `scrollIntoView()` on step targets outside the viewport

```typescript
// useTour.ts (key interface)
const handleJoyrideCallback = useCallback((data: CallBackProps) => {
  const { action, index, type, status } = data;

  if (type === EVENTS.STEP_AFTER) {
    const nextStep = steps[index + 1];
    if (nextStep?.targetRoute && nextStep.targetRoute !== window.location.pathname) {
      setRun(false);
      navigate(nextStep.targetRoute);
      waitForElement(nextStep.target as string).then(() => {
        setStepIndex(index + 1);
        setRun(true);
      });
    } else {
      setStepIndex(index + 1);
    }
    syncToAPI(tourId, index + 1, 'in_progress');
  }

  if (status === STATUS.FINISHED || status === STATUS.SKIPPED) {
    syncToAPI(tourId, steps.length, status === STATUS.FINISHED ? 'completed' : 'skipped');
    setRun(false);
  }
}, [steps, navigate]);
```

### Local storage and API synchronization

Two persistence layers with different latency characteristics:

- **Local storage** is written synchronously on every step change. Read on mount for near-instant restoration.
- **The API** is written asynchronously. It is the source of truth on initial page load and persists across devices.

<Callout type="warning">
Local storage is keyed per tour ID (`lb_onboarding_{tourId}_step`). If the user clears browser storage, the API remains authoritative.
</Callout>

## Cross-Page Navigation {#cross-page-navigation}

Cross-page navigation is the central engineering challenge of this project. React Joyride is designed to spotlight elements that exist in the current DOM. It has no built-in mechanism for navigating to a different route and waiting for new elements to mount before resuming.

### The problem in detail

When the user clicks "Next" on Step 0 (dashboard) and Step 1 needs to spotlight `#connect-services-link` on the Settings page:

1. React Router renders the Settings page component.
2. The Settings page renders its children asynchronously.
3. `#connect-services-link` does not yet exist in the DOM.
4. If Joyride tries to position its tooltip, it errors silently or positions at (0, 0).

### Alternatives considered

| Approach | Why rejected |
|----------|--------------|
| `setTimeout` with fixed delay | Brittle — delay varies by device speed and network conditions. |
| `setInterval` polling | Wastes CPU cycles. Arbitrary interval. Not event-driven. |
| React state sentinel | Requires modifying every target page. Invasive, not scalable. |
| `requestAnimationFrame` loop | Equivalent to polling but faster. Still wastes cycles. |
| **MutationObserver** | Event-driven. Fires exactly when DOM changes. Built into all modern browsers. ✓ |

### The MutationObserver solution

`waitForElement.ts` wraps `MutationObserver` in a `Promise`. It resolves the moment the target element appears in the DOM.

```typescript
export function waitForElement(
  selector: string,
  timeout: number = 10_000
): Promise<Element> {
  return new Promise((resolve, reject) => {
    const existing = document.querySelector(selector);
    if (existing) { resolve(existing); return; }

    let timer: ReturnType<typeof setTimeout>;

    const observer = new MutationObserver((_, obs) => {
      const el = document.querySelector(selector);
      if (el) { clearTimeout(timer); obs.disconnect(); resolve(el); }
    });

    observer.observe(document.body, { childList: true, subtree: true });

    timer = setTimeout(() => {
      observer.disconnect();
      reject(new Error(`"${selector}" not found within ${timeout}ms`));
    }, timeout);
  });
}
```

### Why childList + subtree?

React renders components asynchronously. Observing only direct children of `document.body` misses deeply nested elements like `#connect-services-link` inside a Settings component inside a layout wrapper. `subtree: true` watches for mutations anywhere in the DOM tree.

### Complete cross-page navigation sequence

1. User clicks "Next" on Step 0. Joyride fires callback with `action: NEXT`.
2. `useTour` detects `steps[1].targetRoute !== window.location.pathname`.
3. `setRun(false)` — Joyride pauses. No tooltip shown during navigation.
4. `navigate('/settings/')` — React Router begins route transition.
5. `waitForElement('#connect-services-link')` is awaited.
6. MutationObserver fires as Settings page DOM commits.
7. Promise resolves with element reference.
8. `setStepIndex(1)` and `setRun(true)` — Joyride resumes on Step 1.
9. `scrollIntoView()` called on target element.

### Why pause before navigating?

Without `setRun(false)` first, Joyride stays active during navigation, tries to reposition its tooltip on the old step's target element, and throws a positioning error when that element unmounts.

<Callout type="note">
The 10-second timeout handles navigation failures (e.g., auth redirect). The observer disconnects, the rejected promise is caught, and the tour transitions to `paused` state without leaking memory.
</Callout>

## Feature Guide {#feature-guide}

### What it is

The Feature Guide is a page at `/settings/feature-guide/` that acts as a persistent control center for all available tours. Users see which tours they have completed, which are in progress, and which are still locked.

### The decoupling problem

Feature Guide and TourManager components are in completely separate React component trees. They cannot pass props or callbacks to each other directly.

| Option | Verdict |
|--------|---------|
| Lift state into GlobalAppContext | Pollutes shared context with event-like state |
| URL query parameters | Dirty URLs, requires cleanup |
| **Browser CustomEvents** | Zero shared state, fully decoupled, built-in cleanup ✓ |

### Event registry

| Tour | Start | Restart | Resume |
|------|-------|---------|--------|
| Setup | `lb:start-setup-tour` | `lb:restart-setup-tour` | `lb:resume-setup-tour` |
| Listens | `lb:start-listens-tour` | `lb:restart-listens-tour` | `lb:resume-listens-tour` |
| Stats | `lb:start-stats-tour` | `lb:restart-stats-tour` | `lb:resume-stats-tour` |
| Social | `lb:start-social-tour` | `lb:restart-social-tour` | `lb:resume-social-tour` |

```typescript
// FeatureGuide.tsx

// StatsTourManager.tsx
```

## Challenges {#challenges}

### 1. Cross-page routing

**Problem:** React Joyride spotlights DOM elements. When a step targets an element on a different route, that element doesn't exist in the DOM yet.

**Resolution:** The pause → navigate → MutationObserver → resume pattern in `waitForElement.ts`.

### 2. DOM subtree observation timing

**Problem:** React's async rendering means the route wrapper mounts before its children. Observing only direct children misses deeply nested tour targets.

**Resolution:** `{ childList: true, subtree: true }` on the MutationObserver. The callback runs `querySelector` on every mutation — fast because native selector matching is implemented in C++ in the browser engine.

### 3. Persistent state reconciliation

**Problem:** Two state sources (local storage and the API) can diverge. If an API write fails but local storage was written, the user sees inconsistent state.

**Resolution:** The API is always the authoritative source on initial page load. Local storage is only used as a fast-restore cache while the API response is in flight.

### 4. Viewport-responsive tour targets

**Problem:** The Settings link is in the sidebar on desktop but behind a hamburger menu on mobile. Same tour, different targets.

**Resolution:** `useViewportBreakpoints.ts` + `useMemo` rebuilds the steps array whenever the breakpoint changes.

### 5. Tour interruption handling

**Problem:** If a user navigates away by clicking a link (not the Joyride "Next" button), the tour state becomes stale.

**Resolution:** A React Router navigation listener detects unexpected route changes. On unexpected navigation, the tour is paused and step index is persisted. TourModal prompts resume on return.

### 6. Auto-opening dropdowns

**Problem:** The "Add listens" dropdown is closed by default. Joyride cannot spotlight elements inside a closed dropdown.

**Resolution:** An `onBeforeRun` hook fires before Step 5 and programmatically clicks the dropdown toggle.

### 7. Unlock idempotency

**Problem:** Concurrent GET requests from two tabs could both evaluate unlock conditions and write `unlock_ready = TRUE` simultaneously.

**Resolution:** The database write is idempotent (`UPDATE SET unlock_ready = TRUE` on an already-`TRUE` row is a no-op). Correctness guaranteed at the DB level regardless.

## Lessons Learned {#lessons-learned}

### Read the library source code

React Joyride's documentation describes what it does, not how it does it internally. Understanding that Joyride does a synchronous DOM lookup during its render cycle was essential to building the correct pause-before-navigate sequence.

### DB schema design is load-bearing

The composite primary key and write-once unlock flag were decided in the first design session and survived the entire implementation unchanged. The `INSERT ... ON CONFLICT DO NOTHING` upsert pattern works correctly because of the PK.

### Custom browser events are underrated

Feature Guide ↔ TourManager communication is a real architecture problem. Native browser events (`window.dispatchEvent`) are often the right tool when the pattern is fire-and-forget: zero-dependency, well-understood, easily cleaned up.

### Working in a large open-source codebase

Before writing a single line of code, two weeks were spent reading existing patterns: how hooks are structured, how context providers work, how the SCSS is organized. Building the onboarding system to match these patterns ensures future maintainers can understand and extend it.

### Code review as a learning mechanism

Every PR received detailed code review from MetaBrainz maintainers — covering API design, React patterns, and database concerns. Each comment improved the implementation and built a mental model of production engineering standards.

### MutationObserver is the correct tool for DOM readiness

The instinct when waiting for a DOM element is `setTimeout`. MutationObserver is strictly better: event-driven, no wasted CPU between mutations, resolves exactly when the element exists.

### Progressive unlocking must be enforced server-side

Evaluating unlock conditions on the client would mean the unlock logic is in the browser, where it can be manipulated. Server-side evaluation ensures tours unlock based on actual database state.
